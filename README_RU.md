# NOC Toolkit

**Версия:** 0.5.0
**Единый инструментарий командной строки для NOC операций**

---

## 📖 Обзор

NOC Toolkit — это интерактивный интерфейс командной строки с меню, который предоставляет простой доступ к различным операционным инструментам, используемым командой NOC. Вместо того чтобы запоминать расположение множества скриптов и команд, просто запустите toolkit и выберите нужный инструмент из интерактивного меню.

---

## ✨ Возможности

- 🎯 **Единый интерфейс** - Единая точка входа для всех NOC инструментов
- 📋 **Интерактивное меню** - Простая навигация через меню
- 🔧 **Множество инструментов** - На данный момент включает:
  - **PagerDuty-Jira Tool** - Синхронизация инцидентов PagerDuty с Jira
  - **PagerDuty Job Extractor** - Извлечение названий джоб из смердженных инцидентов PagerDuty
  - **PagerDuty Monitor** - Мониторинг и автоматическое подтверждение triggered инцидентов
  - **PagerDuty Incident Merge** - Поиск и слияние связанных инцидентов по имени джобы
  - **Data Freshness Checker** - Отчёт DACSCAN о свежести данных через Databricks SQL
  - **NOC Report Assistant** - Синхронизация Jira статусов в End-of-Shift Excel отчёт
- 🔐 **Централизованная конфигурация** - Настройте один раз, используйте везде
- 🚀 **Расширяемость** - Легко добавлять новые инструменты
- ✅ **Проверка здоровья** - Автоматически проверяет доступность инструментов

## 🚀 Быстрый старт

### Предварительные требования

- Python 3.7 или выше
- pip (менеджер пакетов Python)

### Установка

1. **Перейдите в директорию toolkit:**
   ```bash
   cd /Users/master/noc-toolkit
   ```

2. **Установите зависимости:**
   ```bash
   pip3 install -r requirements.txt
   ```

3. **Настройте переменные окружения:**

   **ВАЖНО:** Все инструменты используют **единый файл конфигурации** `.env` в корне toolkit!

   ```bash
   # Скопируйте шаблон конфигурации
   cp .env.example .env

   # Отредактируйте .env и заполните ваши учетные данные
   nano .env  # или используйте любой текстовый редактор
   ```

   **Что нужно настроить в `.env`:**

   ```bash
   # PagerDuty API токен (используется обоими инструментами)
   PAGERDUTY_API_TOKEN=ваш_токен_pagerduty

   # Jira конфигурация (для pd-jira-tool)
   JIRA_SERVER_URL=https://jira.livenation.com
   JIRA_PERSONAL_ACCESS_TOKEN=ваш_токен_jira

   # Или для Jira Cloud:
   # JIRA_EMAIL=ваш-email@example.com
   # JIRA_API_TOKEN=ваш_токен_jira_cloud
   ```

### Запуск

Просто запустите toolkit:

```bash
python3 noc-toolkit.py
```

Или сделайте его исполняемым и запускайте напрямую:

```bash
chmod +x noc-toolkit.py
./noc-toolkit.py
```

## 🔧 Доступные инструменты

### 1. PagerDuty-Jira Tool

**Назначение:** Синхронизация инцидентов PagerDuty с задачами Jira

**Возможности:**
- Получение инцидентов PagerDuty по заданным фильтрам
- Автоматическое обнаружение Jira тикетов из заголовков и комментариев инцидентов
- Отслеживание Jira статусов и публикация комментариев с обновлением статуса
- Режим авто-snooze с настраиваемой длительностью
- Автоопределение ключевых слов «ignore»/«disabled» — короткий комментарий и snooze
- 12-часовая защита от дублирования комментариев
- Прогресс-бар с оценкой времени

**Конфигурация:** Использует общий `.env` из корня toolkit (PAGERDUTY_API_TOKEN + Jira credentials)

**Необходимые переменные в `.env`:**
- `PAGERDUTY_API_TOKEN`
- `JIRA_SERVER_URL`
- `JIRA_PERSONAL_ACCESS_TOKEN` (для Jira Server/Data Center)
  ИЛИ
- `JIRA_EMAIL` + `JIRA_API_TOKEN` (для Jira Cloud)

---

### 2. PagerDuty Job Extractor

**Назначение:** Извлечение и анализ графиков дежурств PagerDuty

**Возможности:**
- Получение расписаний PagerDuty
- Извлечение данных о ротации дежурств
- Генерация отчетов о назначенных задачах
- Экспорт данных в различных форматах (CSV, JSON)

**Конфигурация:** См. [tools/pagerduty-job-extractor/README.md](tools/pagerduty-job-extractor/README.md)

**Необходимые переменные в `.env`:**
- `PAGERDUTY_API_TOKEN`

---

### 3. PagerDuty Monitor

**Назначение:** Автоматически продлевает acknowledge инцидентов для предотвращения авто-резолва

**Возможности:**
- Мониторинг acknowledged инцидентов каждые 10 минут (через cron)
- Авто-продление acknowledge до 6-часового таймаута
- Умная логика для избежания спама комментариями
- Отслеживание количества refresh для каждого инцидента
- Оповещение когда нужно ручное обновление (3+ refresh)
- Режим dry-run для безопасного тестирования

**Конфигурация:** См. [tools/pd-monitor/README_RU.md](tools/pd-monitor/README_RU.md)

**Быстрая настройка:**
```bash
cd tools/pd-monitor
# API токен используется из общего .env noc-toolkit
python3 pd_monitor.py --dry-run --verbose  # Сначала тест
```

**Настройка Cron (рекомендуется):**
```bash
crontab -e
# Добавить: */10 * * * * cd /Users/master/pd-monitor && python3 pd_monitor.py >> /tmp/pd-monitor.log 2>&1
```

**Необходимые переменные в `.env`:**
- `PAGERDUTY_API_TOKEN`
- `MONITOR_ACKNOWLEDGE_THRESHOLD_HOURS` (опционально, по умолчанию: 4.0)
- `MONITOR_COMMENT_PATTERN` (опционально, по умолчанию: "working on it")
- `MONITOR_MAX_AUTO_REFRESHES` (опционально, по умолчанию: 3)

### 4. PagerDuty Incident Merge

**Назначение:** Поиск и слияние связанных инцидентов PagerDuty, имеющих одну и ту же первопричину (одно имя джобы/DAG)

**Возможности:**
- Автоматическая группировка инцидентов по нормализованному имени джобы
- Три сценария слияния: same-day (A), cross-date с проверкой через Jira (B), массовый сбой (C)
- Детерминированный выбор целевого инцидента: реальные комментарии > приоритет алерта (Databricks > Monitor > AirFlow) > самый ранний
- Интерактивное подтверждение по группам и по отдельным инцидентам
- Сохранение пропущенных инцидентов между запусками
- Режим dry-run для безопасного предпросмотра

**Быстрая настройка:**
```bash
python3 tools/pd-merge/pd_merge.py --dry-run    # Предпросмотр
python3 tools/pd-merge/pd_merge.py               # Запуск
```

**Опции CLI:**
- `--dry-run, -n` — Симуляция без изменений в API
- `--verbose, -v` — Подробный вывод для отладки
- `--clear-skips` — Очистить список пропущенных инцидентов
- `--show-skips` — Показать текущие пропущенные инциденты

**Необходимые переменные в `.env`:**
- `PAGERDUTY_API_TOKEN`
- `JIRA_SERVER_URL` (опционально, для Сценария B)
- `JIRA_PERSONAL_ACCESS_TOKEN` (опционально, для Сценария B)

### 5. Data Freshness Checker

**Назначение:** Автоматизация ежедневного отчёта DACSCAN Data Freshness через Databricks SQL

**Возможности:**
- Основной отчёт на 15 таблиц (DACSCAN, AGG, AUDIT, SUMMARY, BI-LOADER)
- Автоматические гранулярные проверки для задержанных таблиц (host-level для DACSCAN, max(update_ts) для агрегатных)
- Обработка известной проблемы SALES_ORD_EVENT_OPT (DSSD-29069 — fallback на update_ts)
- Обратный отсчёт SLA (дедлайн 5:30 PM UTC)
- HTML-отчёт с цветовой кодировкой — открывается в браузере для скриншотов в Slack
- Подключение через Databricks SQL Statement Execution REST API (без тяжёлого SDK)

**Быстрая настройка:**
```bash
python3 tools/data-freshness/data_freshness.py --dry-run    # Предпросмотр SQL
python3 tools/data-freshness/data_freshness.py               # Запуск отчёта
python3 tools/data-freshness/data_freshness.py --report      # Запуск + HTML отчёт
```

**Опции CLI:**
- `--report, -r` — Генерация HTML-отчёта и открытие в браузере
- `--check-all` — Гранулярные проверки для ВСЕХ таблиц (не только задержанных)
- `--dry-run, -n` — Показать SQL запросы без выполнения
- `--verbose, -v` — Показать детали API вызовов
- `--format csv/json` — Альтернативные форматы вывода

**Необходимые переменные в `.env`:**
- `DATABRICKS_HOST`
- `DATABRICKS_TOKEN`
- `DATABRICKS_WAREHOUSE_ID`

### 6. NOC Report Assistant

**Назначение:** Синхронизация Jira статусов и добавление строк с тикетами в End-of-Shift Excel отчёт

**Возможности:**
- Синхронизация статусов — обновление Jira статусов (колонка E) для существующих тикетов
- Добавление строки — вставка нового тикета в секцию "Things to monitor" с Jira + Slack ссылками
- Автоопределение Jira и Slack ссылок в любом порядке вставки
- Сохранение всего форматирования Excel, объединений ячеек и гиперссылок
- Работает с обоими листами: Night-Shift-NEW и Day-Shift-NEW

**Быстрая настройка:**
```bash
python3 tools/noc-report-assistant/noc_report_assistant.py --dry-run    # Предпросмотр
python3 tools/noc-report-assistant/noc_report_assistant.py               # Запуск
```

**Опции CLI:**
- `--dry-run, -n` — Показать изменения без сохранения
- `--verbose, -v` — Показать детали API вызовов
- `--file PATH` — Путь к Excel файлу (по умолчанию: `~/Downloads/NOC endshift report.xlsx`)

**Необходимые переменные в `.env`:**
- `JIRA_SERVER_URL`
- `JIRA_PERSONAL_ACCESS_TOKEN`
- `NOC_REPORT_PATH` (опционально)

## 🔐 Безопасность

### Важные замечания по безопасности

- **Никогда не коммитьте файл `.env`** - Он содержит конфиденциальные учетные данные
- **Храните API токены в безопасности** - Сохраняйте их только в переменных окружения или файле `.env`
- **Используйте токены только для чтения, когда возможно** - Ограничьте права токенов до минимально необходимых
- **Ограничьте доступ** - Только авторизованный персонал должен иметь доступ к toolkit

### Права доступа к файлам

Убедитесь, что конфигурационные файлы имеют соответствующие права:

```bash
chmod 600 .env  # Чтение/запись только для владельца
```

## 🔧 Централизованная конфигурация

### Преимущества единого .env файла

✅ **Настройте один раз, используйте везде** - Все инструменты используют один файл .env
✅ **Нет дублирования** - Не нужно копировать токены в несколько мест
✅ **Проще управлять** - Обновите токен в одном месте
✅ **Меньше ошибок** - Снижается риск использования устаревших токенов

### Структура конфигурации

```
noc-toolkit/
├── .env                    # ← Единый файл конфигурации (создайте из .env.example)
├── .env.example            # ← Шаблон со всеми доступными переменными
└── tools/
    ├── pd-jira-tool/      # Использует общий .env из корня
    └── pagerduty-job-extractor/  # Использует общий .env из корня
```

### Как это работает

1. При запуске `noc-toolkit.py` автоматически загружает `.env` из корневой директории
2. Все переменные окружения становятся доступны всем запускаемым инструментам
3. Инструменты автоматически получают доступ к переменным (PAGERDUTY_API_TOKEN, JIRA_*, и т.д.)
4. Не нужно настраивать `.env` в каждой директории инструмента

### Полный пример .env файла

```bash
# ============================================================================
# PagerDuty API Configuration
# ============================================================================
PAGERDUTY_API_TOKEN=your_pagerduty_api_token_here

# ============================================================================
# Jira Configuration (для Jira Server/Data Center)
# ============================================================================
JIRA_SERVER_URL=https://your-jira-server.com
JIRA_PERSONAL_ACCESS_TOKEN=your_jira_personal_access_token_here

# ============================================================================
# Опциональные настройки
# ============================================================================
# SNOOZE_DURATION_HOURS=6.0
# LOG_LEVEL=INFO
# OUTPUT_DIR=./output
```

## 🛠️ Устранение неполадок

### Распространенные проблемы

#### Ошибка "Module not found"

**Проблема:** Python зависимости не установлены

**Решение:**
```bash
pip3 install -r requirements.txt
```

#### "Permission denied" при запуске toolkit

**Проблема:** Скрипт не имеет прав на выполнение

**Решение:**
```bash
chmod +x noc-toolkit.py
```

#### Инструмент показывает [✗] в меню

**Проблема:** Скрипт инструмента не найден или символическая ссылка повреждена

**Решение:**
```bash
# Проверьте директорию tools
ls -la tools/

# Пересоздайте символические ссылки при необходимости
ln -sf /Users/master/pd-jira-tool tools/pd-jira-tool
ln -sf /Users/master/pagerduty-job-extractor tools/pagerduty-job-extractor
```

#### Ошибки аутентификации API

**Проблема:** Неверные или отсутствующие учетные данные

**Решение:**
1. Убедитесь, что файл `.env` существует в корне noc-toolkit
2. Проверьте, что API токены действительны и не истекли
3. Убедитесь, что токены имеют необходимые права доступа
4. Проверьте правильность URL (JIRA_SERVER_URL)

#### Предупреждение "⚠️ Config: No .env file found"

**Проблема:** Файл `.env` не создан

**Решение:**
```bash
cd /Users/master/noc-toolkit
cp .env.example .env
nano .env  # Заполните ваши учетные данные
```

## 🤝 Добавление новых инструментов

Хотите добавить новый инструмент в toolkit?

### Шаг 1: Добавьте инструмент в директорию tools/

```bash
# Вариант А: Скопируйте инструмент
cp -r /путь/к/вашему/инструменту tools/мой-новый-инструмент

# Вариант Б: Создайте символическую ссылку (рекомендуется)
ln -s /путь/к/вашему/инструменту tools/мой-новый-инструмент
```

### Шаг 2: Зарегистрируйте инструмент в noc-toolkit.py

Отредактируйте файл `noc-toolkit.py` и добавьте определение инструмента в метод `_load_tools()` (строки 84-95):

```python
ToolDefinition(
    tool_id="мой-новый-инструмент",
    name="Мой Новый Инструмент",
    description="Описание того, что делает инструмент",
    script_path="tools/мой-новый-инструмент/main.py",
    enabled=True
),
```

### Шаг 3: Добавьте необходимые переменные окружения

Если ваш инструмент требует дополнительных переменных окружения, добавьте их в `.env.example`:

```bash
# Мой Новый Инструмент
# ----------------------------------------------------------------------------
# Описание переменной
MY_TOOL_API_KEY=ваш_ключ_здесь
MY_TOOL_SETTING=значение
```

### Шаг 4: Обновите зависимости

Если инструмент требует дополнительных Python пакетов:

```bash
echo "ваш-пакет>=1.0.0" >> requirements.txt
pip3 install -r requirements.txt
```

### Шаг 5: Протестируйте

```bash
python3 noc-toolkit.py
# Выберите ваш новый инструмент из меню
```
